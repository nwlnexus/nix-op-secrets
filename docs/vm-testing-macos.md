# macOS VM Testing Guide

This guide covers the manual steps for testing the `nix-op-secrets` nix-darwin module
inside a Parallels macOS VM. The Linux (NixOS) path is fully automated via
`./scripts/test-vm.sh`; macOS requires manual VM setup due to Apple's setup assistant.

> **macOS host only.** Linux/Windows contributors cannot run the macOS path
> (Parallels and `aarch64-darwin` are macOS-only). The Linux/Home-Manager path
> is automated end-to-end via `./scripts/test-vm.sh` and is the supported
> testing target for non-macOS contributors.

## Prerequisites

| Requirement | Notes |
|---|---|
| Apple Silicon Mac | aarch64-darwin |
| Parallels Desktop ≥26 | **Pro or Business** edition — `prlctl` is not shipped with Standard |
| Nix on the host | The Linux path runs `nix-shell -p xorriso` to patch the Ubuntu ISO. Determinate installer recommended: `curl --proto '=https' --tlsv1.2 -sSf -L https://install.determinate.systems/nix \| sh -s -- install` |
| macOS installer for the VM | `mist download macos <version> --firmware` (from <https://github.com/ninxsoft/mist-cli>) or download an IPSW from <https://ipsw.me>. `softwareupdate --fetch-full-installer` produces an `.app`, not the IPSW Parallels needs. |
| `op` CLI v2+ | `brew install 1password-cli`; check with `op --version` (must be ≥ 2.0 — v1 syntax is incompatible) |
| `.env` at repo root | See `.env.sample` — token must have access to `nix-op-secrets-test` vault |
| `tests/vm/keys/` | Generated by `setup-base.sh` on the Linux path, or by running the keygen block below |

### What to expect

| Stage | Time | Disk |
|---|---|---|
| First-time base VM build (Linux/Ubuntu path) | ~20 min | ~3 GB cache (ISOs) + ~5 GB base VM |
| Per-cycle test run after base is built | ~3–5 min | ~50 MB linked clone (cleaned up automatically) |
| First macOS VM setup (this guide) | ~30 min | ~30 GB VM image (deep clones; revert via snapshot) |

### Create the test vault and grant service account access

The test suite writes items to a vault named `nix-op-secrets-test`. Your service account
token must have explicit access to this vault — it is **not** granted automatically, even
if the token owner's human account created the vault.

**One-time setup:**

```bash
# 1. Create the vault (as a human user — service accounts cannot create vaults)
op vault create nix-op-secrets-test

# 2. Grant your service account access via the 1Password web UI:
#    a. Sign in at https://start.1password.com
#    b. Go to Developer Tools → Service Accounts → <your service account>
#    c. Under "Vault Access", click "Add vault" → "nix-op-secrets-test"
#       (Read & Write permissions required)

# 3. Verify the token can see the vault:
source .env && op vault get nix-op-secrets-test
```

If step 3 returns an error, the service account has not been granted access yet.

### Generate SSH keys (if not already present)

If you haven't run the Linux test path first, generate keys manually:

```bash
KEY_DIR="$(git rev-parse --show-toplevel)/tests/vm/keys"
mkdir -p "$KEY_DIR"
ssh-keygen -t ed25519 -C "nix-op-secrets-vm-test" -N "" -f "$KEY_DIR/vm_key"
```

## Steps

### 1. Download a macOS IPSW for the VM

`softwareupdate --fetch-full-installer` downloads an Installer **app**, not the
IPSW that Parallels needs. Use `mist-cli` (Homebrew) or ipsw.me:

```bash
# Option A: mist-cli
brew install mist-cli
mist download firmware "15"        # latest macOS 15 IPSW

# Option B: ipsw.me (browser)
#   https://ipsw.me/VirtualMac2,1   ← pick a build, download the .ipsw
```

Note the path of the resulting `.ipsw` — you'll point Parallels at it next.

### 2. Create the VM in Parallels

Open Parallels Desktop → **File → New** → **Install macOS from IPSW** → select the IPSW.

Settings:
- Name: `nix-op-secrets-macos-base`
- CPUs: 4
- Memory: 8 GB
- Disk: 60 GB

### 3. Complete setup assistant

Boot the VM, complete the macOS setup assistant, create user **`nixtest`** with admin privileges.

Enable Remote Login (required for SSH):

```bash
# In the VM's Terminal:
sudo systemsetup -setremotelogin on
```

### 4. Install tooling in the VM

```bash
# In the VM's Terminal:

# Nix (Determinate Systems installer — sets up flakes by default)
curl --proto '=https' --tlsv1.2 -sSf -L https://install.determinate.systems/nix \
  | sh -s -- install

# 1Password CLI
brew install 1password-cli
```

> No need to install nix-darwin manually inside the VM. Step 9 below invokes
> `nix run github:nix-darwin/nix-darwin -- switch --flake ...` which fetches
> `darwin-rebuild` from the flake and uses our test configuration directly —
> no separate bootstrap is required.

### 5. Authorise SSH from host

From the **host machine** (after SSH keys have been generated):

```bash
SSH_OPTS="-o StrictHostKeyChecking=no -o UserKnownHostsFile=/dev/null"
VM_IP=<check Parallels UI or prlctl list -o ip>

ssh-copy-id -i tests/vm/keys/vm_key.pub nixtest@"$VM_IP"
# Alternatively, append tests/vm/keys/vm_key.pub to ~/.ssh/authorized_keys in the VM
```

Test connectivity:

```bash
ssh -i tests/vm/keys/vm_key $SSH_OPTS nixtest@"$VM_IP" echo "SSH OK"
```

### 6. Snapshot the base state

```bash
prlctl snapshot nix-op-secrets-macos-base --name "clean"
```

### 7. Create test 1Password items

From the **host machine** (with `.env` sourced):

```bash
source .env

VAULT="nix-op-secrets-test"

FIELD_ID=$(op item create \
  --vault "$VAULT" --category Login \
  --title "nix-op-secrets-test-field" \
  "password=test-field-value" \
  --format json | jq -r '.id')

SSH_ID=$(op item create \
  --vault "$VAULT" --category "SSH Key" \
  --title "nix-op-secrets-test-ssh" \
  --format json | jq -r '.id')

DOC_ID=$(op document create tests/vm/fixtures/test-doc.txt \
  --vault "$VAULT" \
  --title "nix-op-secrets-test-doc" \
  --format json | jq -r '.id')

echo "FIELD_ID=$FIELD_ID  SSH_ID=$SSH_ID  DOC_ID=$DOC_ID"
# Save these — needed for cleanup in step 11
```

### 8. Inject service account token into VM

The token is sent over SSH stdin so it never appears on the command line
(visible to `ps`) or expands shell metacharacters.

```bash
SSH_OPTS="-i tests/vm/keys/vm_key -o StrictHostKeyChecking=no -o UserKnownHostsFile=/dev/null"
source .env

printf '%s' "$OP_SERVICE_ACCOUNT_TOKEN" | ssh $SSH_OPTS nixtest@"$VM_IP" \
  "sudo tee /etc/op-secrets-test-token >/dev/null \
    && sudo chown nixtest:nixtest /etc/op-secrets-test-token \
    && sudo chmod 600 /etc/op-secrets-test-token"
```

### 9. Mount repo and run darwin-rebuild

```bash
# Mount the repo as a Parallels shared folder
prlctl set nix-op-secrets-macos-base \
  --shf-host-add nix-op-secrets \
  --shf-host-path "$(pwd)"
# The repo is now available at /Volumes/nix-op-secrets inside the VM

# Run darwin-rebuild switch from the host over SSH
# --override-input redirects op-secrets to the mounted checkout without
# modifying the committed flake.lock
ssh $SSH_OPTS nixtest@"$VM_IP" \
  "sudo darwin-rebuild switch \
    --flake /Volumes/nix-op-secrets/tests/vm#test-macos \
    --override-input op-secrets path:/Volumes/nix-op-secrets \
    --no-write-lock-file"
```

### 10. Run assertions

> **Note:** macOS uses `stat -f '%OLp'` (BSD stat) rather than `stat -c '%a'` (GNU stat used in the Linux scripts). Use the commands below as-is.

Same checks as the Linux automated path, executed over SSH:

```bash
SSH="ssh $SSH_OPTS nixtest@$VM_IP"
fail()          { echo "FAIL: $1" >&2; exit 1; }
check_file()    { $SSH "test -f '$1'" || fail "missing: $1"; }
check_mode()    { MODE=$($SSH "stat -f '%OLp' '$1'"); [[ "$MODE" == "$2" ]] || fail "mode $MODE != $2 on $1"; }
check_content() { $SSH "grep -qF '$2' '$1'" || fail "expected '$2' in $1"; }

check_file    "/Users/nixtest/.local/secrets/field.txt"
check_mode    "/Users/nixtest/.local/secrets/field.txt" "600"
check_file    "/Users/nixtest/.ssh/test-vm-key"
check_mode    "/Users/nixtest/.ssh/test-vm-key" "600"
check_file    "/Users/nixtest/.ssh/test-vm-key.pub"
check_mode    "/Users/nixtest/.ssh/test-vm-key.pub" "644"
check_file    "/Users/nixtest/.local/secrets/doc.txt"
check_mode    "/Users/nixtest/.local/secrets/doc.txt" "600"
check_file    "/Users/nixtest/.local/secrets/infra.env"
check_mode    "/Users/nixtest/.local/secrets/infra.env" "600"
check_content "/Users/nixtest/.local/secrets/infra.env" "test-field-value"

for KEY in test-field test-ssh "test-ssh__pub" test-doc test-template; do
  $SSH "jq -e '.\"$KEY\"' /Users/nixtest/.local/state/op-secrets/manifest.json" > /dev/null \
    || fail "manifest missing: $KEY"
done

echo "All macOS VM integration tests passed"
```

### 11. Cleanup

```bash
# Revert VM to clean snapshot (ready for next test run)
prlctl snapshot-switch nix-op-secrets-macos-base --id "$(prlctl snapshot-list \
  nix-op-secrets-macos-base -j | jq -r 'to_entries[]|select(.value.name=="clean")|.key')"

# Delete test 1Password items
source .env
op item delete "$FIELD_ID" --vault nix-op-secrets-test
op item delete "$SSH_ID"   --vault nix-op-secrets-test
op item delete "$DOC_ID"   --vault nix-op-secrets-test
```

## Troubleshooting

**SSH key auth rejected.** The autoinstall (Linux path) injects the host's
public key via subiquity's `ssh.authorized-keys`. If that fails, check the VM
console: `prlctl capture nix-op-secrets-base --file /tmp/vm.png` and open
`/tmp/vm.png`. The most common cause is a missing `autoinstall` kernel
parameter — `setup-base.sh` patches this into the GRUB config of the cached
ISO; delete `.cache/ubuntu-*-autoinstall.iso` to force a rebuild.

**`prlctl list … -o ip` is empty even though the VM is up.** Parallels needs
1–15 minutes to surface a DHCP lease via `prlctl` on first boot. `setup-base.sh`
waits up to 1800 s. If a run times out, the VM probably *did* get an IP — check
manually with `prlctl list nix-op-secrets-base -o ip --no-header` after a minute.

**Stuck or orphan VM.** Force-cleanup with:

```bash
prlctl stop   nix-op-secrets-base --kill 2>/dev/null
prlctl delete nix-op-secrets-base
rm -rf .cache/autoinstall-seed.iso             # forces fresh seed ISO
```

**Home-Manager activation log.** Inside the VM, look in
`/home/nixtest/.local/state/op-secrets/` for `manifest.json` (what's been
written) and the systemd journal for `home-manager` units. The activation
script writes to stderr/stdout, captured in the `nix run home-manager …`
output during a switch.

**Another run is in progress.** `run-tests.sh` acquires
`/tmp/nix-op-secrets-vm-test.lock` to prevent concurrent runs (which would
race on the shared 1Password vault). Remove the directory manually if a
previous run died without cleanup.

## CI

The Linux automated path (`scripts/test-vm.sh`) is **macOS-host only** because
it drives Parallels. To run in CI you need a self-hosted macOS runner with
Parallels Desktop installed and `prlctl` on PATH; GitHub-hosted runners do not
support nested virtualisation for Parallels.

The service account token **must** be stored as a masked secret. With GitHub
Actions:

```yaml
- name: Run VM integration tests
  env:
    OP_SERVICE_ACCOUNT_TOKEN: ${{ secrets.OP_SERVICE_ACCOUNT_TOKEN }}
  run: |
    printf 'OP_SERVICE_ACCOUNT_TOKEN=%s\n' "$OP_SERVICE_ACCOUNT_TOKEN" > .env
    ./scripts/test-vm.sh
```

**Never** hardcode the token in workflow YAML or commit it to `.env`. The
`.gitignore` already excludes `.env`; verify before pushing.
